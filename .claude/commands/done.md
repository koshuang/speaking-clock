# 任務完成後的文件維護

請根據以下流程執行任務完成後的文件更新：

## 步驟

1. **詢問變更摘要** - 請用戶描述完成了什麼工作
2. **分析變更類型** - 判斷屬於哪種類型的變更
3. **評估文件更新需求** - 根據下方表格判斷需要更新哪些文件
4. **執行文件更新** - 更新必要的文件
5. **顯示更新摘要** - 告知用戶更新了哪些內容

## 變更類型與文件對應

| 變更類型 | README.md | PRD.md | AGENT.md | 專屬文件 |
|----------|:---------:|:------:|:--------:|:--------:|
| 新增使用者可見功能 | ✅ 功能特色 | ✅ 功能需求 | - | - |
| 新增技術元件/API | ✅ 技術棧 | ✅ 技術規格 | - | 視情況 |
| 修改架構或目錄結構 | ✅ 專案架構 | - | ✅ 專案結構 | - |
| 新增/修改測試 | ✅ 測試章節 | - | - | - |
| 新增設定選項 | - | ✅ 功能需求表 | - | - |
| 新增 UI 元件庫或樣式系統 | ✅ 技術棧 | ✅ 技術選型 | - | - |
| 新增需要設定的外部服務 | ✅ 文件連結 | ✅ 技術選型 | ✅ 擴展指引 | ✅ 專屬設定指南 |
| Bug 修復 | - | - | - | - |
| 重構（不改變功能） | - | - | 視情況 | - |

## 文件更新原則

### 核心文件
- **README.md**: 面向使用者和新開發者，**保持簡潔**，複雜內容用連結指向專屬文件
- **docs/PRD.md**: 產品規格文件，記錄功能需求和技術規格
- **AGENT.md**: AI 開發指引，記錄架構和開發規範

### 專屬文件原則

當變更涉及以下情況時，應建立專屬文件而非塞入 README：

| 情況 | 建議做法 |
|------|----------|
| 外部服務設定（如 Supabase、Firebase） | 建立 `docs/{SERVICE}-SETUP.md` |
| 複雜的開發環境設定 | 建立 `docs/DEVELOPMENT.md` |
| API 文件 | 建立 `docs/API.md` |
| 部署指南超過 10 行 | 建立 `docs/DEPLOYMENT.md` |

**README 只放連結**：
```markdown
## 文件
| [Supabase 設定指南](./docs/SUPABASE-SETUP.md) | 雲端同步功能設定 |
```

### 現有專屬文件
